# FADE — 프로젝트 기획서 (Claude Code용)

> 이 문서는 Claude Code가 프로젝트 전체 컨텍스트를 파악하기 위한 단일 진입점입니다.
> 코드 작성 전 반드시 전체를 읽고, 모호한 부분은 사용자에게 질문한 뒤 진행하세요.

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| **서비스명** | FADE |
| **한 줄 설명** | 사후 행정 및 상속 도움 플랫폼 |
| **핵심 가치** | 장례 및 상속을 처음 겪는 사용자가 전문가 없이도 자신의 상황에 맞는 행정 처리 및 절세 방법을 파악하도록 돕는 AI 기반 웹 서비스 |
| **팀** | Focus on |
| **산출물** | 사망 전 재산 정리 및 절세 방법 안내 + 사망 후 처리할 행정·재무·생활 과제를 시계열로 제시 + 상속 정보 기반 세금 계산 및 절세 시뮬레이터 |

---

## 2. 타겟 사용자

### Primary (B2C)
- 가족의 사망으로 **상속 절차를 밟는 성인**
- 세무·법률 지식이 거의 없음
- 세무사 선임 비용이 부담스러운 중산층

### Secondary (B2B)
- 세무사, 법무사, 변호사 등 **상속 관련 전문가**
- 고객 발굴 채널 + 업무 도구가 필요한 개인 사무소

---

## 3. 해결하려는 문제

1. **정보 파편화** — 사망 후 접촉해야 할 기관이 17개 이상, 정보가 흩어져 있음
2. **법정 기한 놓침** — 한정승인(3개월), 상속세 신고(6개월) 등을 몰라서 가산세 발생
3. **세무사 비용 부담** — 비용 때문에 절세 전략 없이 과납하는 사례 다수
4. **생활 행정 공백** — 기존 세무/법무 플랫폼은 신고·분쟁에만 특화, 일상 행정 안내 없음

---

## 4. 핵심 기능 (MVP 스코프)

### F1. 사후 처리 타임라인 & 체크리스트 🟢 무료
- **데이터**: 사망일 기준 D+N일 시계열 할 일 목록
- **핵심 마감일**:
  - D+3개월: 한정승인·상속포기 신고 기한
  - D+6개월: 상속세 신고·납부 기한
- **기관별 체크리스트**: 주민센터, 국세청, 은행, 보험사, 통신사 등 17개 이상
- **진행 상황 저장·추적** (로그인 사용자)

### F2. 상속 정보 입력 위저드 🟢 무료
- **단계별 입력**:
  1. 사망일
  2. 가족관계 (배우자 유무, 자녀 수, 기타 상속인)
  3. 재산 종류별 입력 (부동산 / 금융자산 / 기타)
- **OCR 자동 입력** — 등기부등본·통장 사본 업로드 시 자동 인식 (Phase 2)
- **입력값 저장 → 타 기능으로 공유**

### F3. 상속세 자동 계산 엔진 🟢 무료 (핵심)
- 재산 유형별 입력값 기반 **자동 계산**
- **법정 공제 자동 적용**
  - 일괄공제 5억 원
  - 배우자 공제 (최소 5억 ~ 최대 30억)
  - 금융재산 공제, 동거주택 상속공제 등
- **누진세율 구간 시각화** (10%~50%)
- **예상 납부세액 실시간 출력** — 입력값 변경 시 즉시 갱신
- ⚠️ 정확성 중요: 국세청 자료 기준으로 구현, 단위 테스트 필수

### F4. 절세 시뮬레이터 🟢 무료
- 여러 시나리오를 **나란히 비교**
  - 예: "배우자 단독 상속" vs "자녀 균등 분할" vs "배우자 50% + 자녀 50%"
- 시나리오 저장 (로그인 사용자)
- 시나리오 공유 링크 생성

### F5. AI 절세 전략 추천 🟡 유료 전환 포인트
- **LLM 기반** (Anthropic Claude API 사용)
- 사용자 입력 데이터 → 맞춤형 절세 전략 3개 생성
- 공제 극대화 방법, 분할 상속 전략, 증여 시점 조언 등
- **Q&A 챗봇**: 상속세·증여세 질의응답
- **세무사 연계 필요 시점 안내** (서비스 한계 명확히 고지)

### F6. 세무사 검토 리포트 🔴 유료 (1회 결제)
- AI 생성 초안 → **세무사 검토·수정·최종 승인** → 사용자에게 PDF 전달
- 세무사 매칭 (지역·전문 분야 기준)
- 결제 완료 후 전문가에게 전달되는 워크플로우

### F7. 전문가 포털 (B2B) 🔴 구독형
- **별도 로그인 영역** (/pro)
- 할당된 유족의 자산·부채 리포트 열람
- 검토 의견 작성·승인 워크플로우
- 프로필 관리 (광고 노출용)

## 5. 페이지 구조 & 라우팅

> Next.js App Router 기준. `(group)` 은 URL에 영향 없는 그룹핑.

```
app/
├── (marketing)/
│   └── page.tsx                    # / — 랜딩 (서비스 소개, 문제 인식, CTA)
│
├── (onboarding)/
│   └── onboarding/
│       ├── step-1-date/page.tsx    # 사망일 입력
│       ├── step-2-family/page.tsx  # 가족관계 입력
│       └── step-3-assets/page.tsx  # 재산 입력
│
├── (app)/
│   ├── dashboard/page.tsx          # 타임라인 + 체크리스트 + 진행률
│   ├── calculator/page.tsx         # 상속세 계산기
│   ├── simulator/page.tsx          # 절세 시뮬레이터 (시나리오 비교)
│   ├── chat/page.tsx               # AI 상담 챗봇
│   ├── report/[id]/page.tsx        # 리포트 상세 (유료)
│   ├── experts/page.tsx            # 전문가 매칭 목록
│   ├── experts/[id]/page.tsx       # 전문가 프로필
│   └── me/page.tsx                 # 마이페이지 (결제, 저장된 시나리오)
│
├── (pro)/
│   └── pro/
│       ├── login/page.tsx          # 전문가 로그인
│       ├── dashboard/page.tsx      # 전문가 대시보드
│       └── cases/[id]/page.tsx     # 할당된 유족 케이스 상세
│
├── api/
│   ├── calculator/route.ts         # 상속세 계산 API
│   ├── ai/strategy/route.ts        # LLM 절세 전략 생성
│   ├── ai/chat/route.ts            # 챗봇
│   ├── ocr/route.ts                # OCR 처리
│   └── payment/route.ts            # 결제 웹훅
│
└── layout.tsx                      # 공통 레이아웃
```

## 6. 데이터 모델 (DB 스키마 초안)

### users
- `id`, `email`, `name`, `role` (user / pro / admin), `created_at`

### cases (유족의 상속 건 하나)
- `id`, `user_id` (FK), `deceased_name`, `death_date`, `created_at`, `updated_at`

### heirs (상속인)
- `id`, `case_id` (FK), `relationship` (spouse / child / parent / sibling), `name`

### assets (재산)
- `id`, `case_id` (FK), `type` (real_estate / financial / other), `value`, `description`

### checklists
- `id`, `case_id` (FK), `task_key`, `due_date`, `completed`, `completed_at`

### scenarios (절세 시뮬레이션 시나리오)
- `id`, `case_id` (FK), `name`, `allocation` (JSON), `calculated_tax`, `created_at`

### reports (세무사 검수 리포트)
- `id`, `case_id` (FK), `expert_id` (FK), `status` (draft / reviewing / approved), `content`, `paid_at`

### experts (전문가)
- `id`, `user_id` (FK), `specialty`, `region`, `profile`, `subscription_tier`

## 8. 기술 스택

### 확정
- **프론트엔드**: Next.js 15 (App Router) + TypeScript
- **스타일**: Tailwind CSS + shadcn/ui
- **배포**: Vercel

### 권장 (변경 가능 — 결정 필요)
- **DB & Auth**: Supabase (PostgreSQL + Auth + Storage 통합)
- **AI**: Anthropic Claude API (절세 전략 생성, 챗봇)
- **OCR**: Naver CLOVA OCR (한국 서류 특화) 또는 Google Cloud Vision
- **결제**: 토스페이먼츠
- **상태관리**: Zustand 또는 React Context (온보딩 위저드용)
- **폼**: React Hook Form + Zod
- **테스트**: Vitest (단위) + Playwright (E2E)

## 11. Claude Code 작업 가이드

### 작업 시작 시 항상
1. 이 `PLAN.md`와 `CLAUDE.md`(있다면)를 먼저 읽기
2. 관련 페이지/API의 기존 코드를 읽고 컨벤션 확인
3. 모호한 요구사항은 **추측하지 말고 질문**

### 코딩 원칙
- **TypeScript strict 모드** — `any` 금지
- **컴포넌트 단위 분리** — 한 파일 300줄 초과 금지
- **서버 컴포넌트 우선** — 인터랙션 필요할 때만 `"use client"`
- **세법 관련 상수는 중앙화** — `lib/tax/constants.ts` 한 곳에서 관리
- **법정 기한·공제액 변경 대비** — 하드코딩 최소화, 설정 파일로 분리

### 세법 관련 주의사항 ⚠️
- 상속세 계산 결과는 **법적 효력이 없음**을 UI에 명시
- 모든 계산 로직은 **국세청 공식 자료 근거 주석** 필수
- 계산 엔진은 **순수 함수**로 작성 + **단위 테스트 커버리지 90% 이상**

### 커뮤니케이션
- 큰 구조 변경 전에는 계획을 먼저 제시하고 승인받기
- 외부 API 키가 필요하면 `.env.example`에 항목 추가하고 알려주기
- 법률·세무 관련 문구는 초안을 작성한 뒤 "세무사 자문 필요" 주석 달기

## 12. 용어집 (도메인 지식)

| 용어 | 설명 |
|---|---|
| 상속세 | 사망으로 인한 재산 이전에 부과되는 세금 |
| 증여세 | 생전에 재산을 무상 이전할 때 부과되는 세금 |
| 한정승인 | 상속받은 재산 한도 내에서만 채무 변제 — **사망일로부터 3개월 내 신고** |
| 상속포기 | 상속 자체를 거부 — **사망일로부터 3개월 내** |
| 일괄공제 | 상속세 계산 시 기본적으로 공제되는 5억 원 |
| 배우자 공제 | 배우자 상속분에 대한 공제 (최소 5억 ~ 최대 30억) |
| 안심상속 원스톱 서비스 | 정부가 제공하는 사망자 재산 조회 서비스 (경쟁·보완 관계) |
